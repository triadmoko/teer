## What this is

**Teer** — a Terminal Workspace Manager desktop app. Group CLI sessions (dev servers, watchers, SSH, etc.) by project workspace; each session is a full PTY rendered with xterm.js. Built with **Wails v3** (Go backend + Svelte 5 / TypeScript frontend, OS WebView — no bundled Chromium).

> Module and binary are named `teer` (lowercase); "Teer" is the display name.

## Commands

All dev tasks go through [Task](https://taskfile.dev) (`Taskfile.yml`), which delegates to per-OS files in `build/<os>/Taskfile.yml`.

```bash
task dev            # hot-reload dev (Vite + Go), via `wails3 dev`
task build          # production build → bin/
task run            # run production build
task package        # package for distribution
task build:server   # headless HTTP server build (-tags server), no GUI deps
task run:server     # run server-mode binary
task build:docker   # distroless image for server mode
```

Go tests (standard toolchain — no Task wrapper):

```bash
go test ./...                              # all
go test ./internal/service/...             # one package
go test -run TestName ./internal/service   # single test
```

Frontend checks (run from `frontend/`):

```bash
npm run check       # svelte-check type checking
```

**Wails bindings**: `frontend/bindings/` is generated by `wails3 generate bindings` (auto-run during `task build`/`dev`). Do NOT hand-edit — change the Go service signature, then regenerate. Exported methods on registered services become callable from TS.

## Architecture

### Backend (Go) — hexagonal / ports-and-adapters

- `internal/domain/` — pure types (`Workspace`, `SessionDef`, `Config`) and interfaces (`Repository`, `EventEmitter`). No external deps.
- `internal/infra/` — adapters: `config/store.go` (JSON persistence at `~/.config/teer/config.json`, atomic write via tmp+rename, `0600`), `terminal/` (PTY abstraction; `pty_unix.go` uses `creack/pty` behind `//go:build linux || darwin`, `pty_windows.go` for ConPTY).
- `internal/service/` — application logic, registered as Wails services in `main.go`: `SessionService`, `WorkspaceService`, `DialogService`, `UpdaterService`.

`main.go` wires everything: builds the `config.Store`, registers the four services, opens the window. `Version` is injected at build time via `-ldflags "-X main.Version=..."`.

### Terminal I/O flow (the core data path)

Keystrokes and output do NOT go through normal request/response — output is **streamed via Wails Events**:

```
xterm.js --WriteSession(id,data)--> SessionService --> PTY --> shell
xterm.js <--Events.On("session:<id>:out")-- readLoop goroutine <-- PTY
```

- `SessionService.StartSession` spawns a PTY and launches a `readLoop` goroutine per session. The goroutine reads PTY bytes, **base64-encodes** them, and emits `session:<id>:out`. On exit it emits `session:<id>:exit` with code/err.
- Output is base64 because event payloads are JSON — raw terminal bytes aren't UTF-8 safe. Frontend decodes in `infrastructure/wails/terminal-transport.ts` (`b64ToBytes`).
- `live` sessions map is mutex-guarded; `ServiceShutdown` kills all PTYs on quit (sessions are NOT persisted across restarts — only their `SessionDef` definitions are).

### Frontend (Svelte 5) — layered, mirrors the backend

Path aliases defined in `svelte.config.js`. Layers (outer depends inward):

- `src/domain/` — frontend models (`models.ts`, `layout.ts`, `terminalSettings.ts`).
- `src/infrastructure/wails/` — thin wrappers over generated `@bindings`. **All Wails calls go through here**, re-exported from its `index.ts` (`workspaceRepository`, `sessionGateway`, `terminalTransport`, `updaterGateway`). UI never imports `@bindings` directly.
- `src/application/` — state + use-cases. Svelte stores live in `stores.ts` (`workspaces`, `activeWorkspaceId`, `activeSessionId`, `running`, ...); one module per feature (session, workspace, terminal, commandPalette, updater, dialogs). Barrel-exported via `index.ts` (`@application`).
- `src/presentation/components/` — Svelte UI (`Terminal.svelte`, `Sidebar.svelte`, `TabBar.svelte`, `CommandPalette.svelte`, dialogs).
- `src/routes/` — SvelteKit pages. Uses **adapter-static** with SPA fallback (`index.html`); no SSR — the app ships as static assets embedded in the Go binary (`//go:embed all:frontend/dist`).

### Data model

```
Config → Workspace[] (name, color, defaultCwd, startupCommand, env)
            └── SessionDef[] (name, shell, cwd, env, autoStart, layout)
```

`WorkspaceService` is the single owner of persistence: every mutation does `repo.Load()` → modify → `repo.Save()`. There is no in-memory cache of the config in the service layer.

## Conventions

- **Comments and error messages are in Indonesian** (e.g. `errors.New("workspace: nama wajib diisi")`). Match this when editing existing Go code.
- Platform-specific Go files use build tags (`pty_unix.go` / `pty_windows.go`); keep the `PTY` interface in `pty.go` as the shared contract.
- Adding a backend feature callable from the UI: add the method to a service in `internal/service/`, regenerate bindings, then wrap it in `infrastructure/wails/` before using it in `application/`.
